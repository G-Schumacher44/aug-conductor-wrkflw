Welcome. Conductor is a structured agent workflow system — a scaffold you drop into any repo to give AI agents durable context, structured execution, and safe handoffs between sessions. Today: what it is, where it came from, and three live demos.

The Problem

Agents forget.

  • Every new session starts cold — no memory of prior work
  • No structured handoff between sessions or agents
  • No scheduling mechanism — "what comes next?"
  • Result: meta-work, re-explanation, scope drift
The core problem with AI agents today: they're stateless. Every session you re-explain the project. The agent re-discovers the codebase. Decisions made last session aren't recorded. The result is that the human does a lot of meta-work that shouldn't be their job.

The Conductor Loop

State lives in files, not in the model.

intent defined
↓
slice executed
↓
handoff written
↓
Exact Next Steps
↓
operator approves
↓
repeat
The solution: treat the filesystem as persistent memory. Every agent session reads a structured spine — the intent, the active slice, the handoff. It executes bounded work. It writes a handoff with Exact Next Steps. The next session picks up from the handoff. No state lives in the agent's memory. The handoff IS the scheduling mechanism.

What Is a Slice?

A bounded unit of work:

  • Explicit scope + gates
  • Out-of-scope declarations
  • Acceptance criteria (checkboxes)
  • One commit or PR
  • Completable in a single session
slice-04-promotions-view.md

objective:
  add fct_promotions view
  and 9th explore

out of scope:
  - field enrichment (slice-05)
  - backfill logic

gate:
  python3 scripts/validate.py

status: active
A slice is the atomic unit of work. Small enough that one agent session completes it. It declares exactly what's in scope, what's explicitly out, and what the validation gate is. When done, the agent marks it stable, advances the queue, and writes a handoff. The agent never invents next steps from scratch — the Exact Next Steps in the handoff do that.

The Scaffold

Drop into any repo · fill intent.md · point an agent at it.

conductor/
  index.md              ← active slice pointer + queue
  master-plan-*.md      ← full project scope + phases
  slice-NN-*.md         ← active execution artifact
  handoff-log.md        ← current-state handoff only
  handoff-archive.md    ← older entries archived here
  tracks.md             ← cross-repo dependencies
  AGENTS.md             ← rename to CLAUDE.md or GEMINI.md

intent.md               ← fill this in first
scripts/
  validate.py           ← governance gate (zero deps)

Everything else — handoff format, queue advancement, archive pattern — is identical regardless of domain.

This is the full scaffold. Copy it into any repo. Fill in intent.md with your project goal, stack, and first milestone. Edit slice-01 for your first unit of work. Everything else runs the same whether you're building LookML, a REST API, a data pipeline, or anything else.

Conductor traces to Google — released December 17, 2025 by the Gemini CLI team. The core idea: keep project state in a conductor/ directory so agents can orient without session context. Powerful foundation. The filesystem-as-state principle itself traces to Anthropic's CLAUDE.md convention: put what the agent needs in files, not in the model's memory. This repo extends the foundation with slices, a structured handoff mechanism, multi-agent support, a validation gate, and cross-repo coordination.

Demo 1 — Greenfield Bootstrap

Branch: main · Entry: DEMO.md · ~5 min

  • Pre-deployed project/conductor/ spine — 3 slices queued
  • Agent reads spine → branches → generates 8 LookML views
  • One scoped commit per view
  • Generates model with 8 explores
  • Runs python3 scripts/validate.py — required gate
  • Marks slice stable · advances queue · writes handoff

No scaffolding, no meta-work. First prompt → first commit → first handoff.

Demo 1 is the greenfield case. The conductor spine is pre-deployed — the agent doesn't scaffold it, it uses it. One prompt, and it immediately starts executing real work. By the end: 8 view files, a model file, a passing validator, and a handoff with Exact Next Steps ready for operator review.

Demo 2 — Pair Programming

Branch: demo-2-start · Entry: DEMO2.md · ~8 min

Phase 1 — Autonomous

  • Reads slice-04
  • Generates fct_promotions view
  • Adds 9th explore to model
  • Validates → writes handoff
  • Exact Next Steps → slice-05

Phase 2 — Collaborative

  • Reads Exact Next Steps
  • Drafts slice-05 spec
  • Commits draft → stops
  • Operator reviews diff in IDE
  • Approves → agent executes

Agent proposes · Operator approves · Agent executes

Demo 2 shows the pair programming pattern. Phase 1 is fully autonomous — the agent picks up slice-04 and runs it. Then it writes a handoff with Exact Next Steps for slice-05. Phase 2: the agent reads those steps, drafts the slice-05 spec, commits it, and stops. The operator reviews the diff live in the IDE — adjusts scope or wording — then approves. The agent executes. This is the core collaboration model.

Demo 3 — Automated Maintenance

Branch: demo-3-start · Entry: DEMO3.md · ~2 min

You are an automated maintenance agent.
A cron job fired you — no human is present.
You have no session context.
Orient yourself from the Conductor spine only.
  • Runs python3 scripts/validate.py
  • Verifies all slices STABLE in index.md
  • Checks handoff-log.md for open blockers
  • Writes status: clean or degraded
  • Commits and exits

A well-formed Conductor spine is self-describing.
Any agent, any time, can orient from it alone.

Demo 3 simulates a nightly cron job. No human, no session context. The agent wakes up cold, reads the spine, runs the validator, checks for open blockers, and writes a structured maintenance report. Status is clean or degraded — it does not auto-fix. Commits and exits. This is the proof that a well-formed Conductor spine is self-describing.

Validator

scripts/validate.py — core

  • Spine structure present
  • Handoff has Commit: + Exact Next Steps
  • Active slice acceptance criteria
  • CI stub present
  • Git branch state

demo/scripts/validate_lookml.py — extension

  • manifest.lkml + project_name
  • View blocks + dimensions + measure
  • Model connection + explores
$ python3 scripts/validate.py

✓ conductor/index.md present
✓ conductor/handoff-log.md present
✓ active slice: none — all slices stable
✓ CI workflow stub
✓ Git branch: feat/slice-01

All checks passed.

stdlib only · zero deps · CI-safe · extensible

The validator is the governance gate — never optional. scripts/validate.py is domain-agnostic: spine structure, handoff format, active slice, git branch. Domain-specific checks are separate extension scripts. For this demo, validate_lookml.py handles LookML structural checks. Same pattern works for any domain — copy it and replace the checks.
