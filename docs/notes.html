Conductor — Speaker Notes

conductor · structured agent workflow · GS Analytics

01 Title

Welcome. Conductor is a structured agent workflow system — a scaffold you drop into any repo to give AI agents durable context, structured execution, and safe handoffs between sessions. Today I'll walk through what it is, where it came from, and three live demos that run from one prompt each. Total demo runtime is about fifteen minutes if you want to run all three back-to-back.

02 The Problem

The core problem with AI coding agents today is that they're stateless. Every session you re-explain the project. The agent re-discovers the codebase. Decisions made last session aren't recorded anywhere durable. The result is that the human becomes the memory layer — doing meta-work, re-orientation, reconciling context. That's not where the operator's leverage should be.

03 Without / With Conductor

Conductor's solution is to treat the filesystem as persistent memory. Every agent session reads a structured spine — the intent, the active slice, the latest handoff. It executes bounded work. It writes a new handoff. The next session picks up from there. State is independent of any one model context, any one agent, any one session. That's the whole pattern.

04 The Loop

This is the loop. The operator defines intent. The agent executes a slice. The agent writes a handoff with Exact Next Steps. The operator reviews and approves or redirects. The queue advances. Repeat. The handoff is the scheduling mechanism — the agent proposes what comes next, the operator decides whether to promote it to the next slice spec. Agent proposes, operator approves, agent executes.

05 What Is a Slice

A slice is the atomic unit of work in this system. Small enough that one agent session completes it. It declares exactly what's in scope and what's explicitly out — scope drift is the default failure mode and the slice format kills it. Acceptance criteria are real checkboxes, not vibes. And the validation gate is non-optional — no handoff is written until the validator passes.

06 The Scaffold

This is the entire scaffold. Twelve files. No service, no database, no API. The conductor directory is identical regardless of domain — LookML, REST APIs, dbt pipelines, frontend work, all the same format. You fill in intent.md, you edit slice-01 for your first piece of work, you point an agent at the repo root with one sentence. That's it.

07 Origins

Conductor as a pattern traces to Google — released December 17, 2025 by the Gemini CLI team. The core idea — keep project state in a conductor directory so agents can orient without session context — is their contribution. The filesystem-as-state principle itself goes further back to Anthropic's CLAUDE.md convention. This repo extends that foundation with slice-based execution, the Exact Next Steps handoff, multi-agent support, the validation gate, and cross-repo coordination.

08 Three Demos

Three demos, three branches, fully independent — you don't need to run demo one before demo two. Each branch is pre-seeded with the right starting state. Greenfield bootstrap on main. Iterative feature with pair programming on demo-2-start. Automated maintenance on demo-3-start. One prompt per demo: read DEMO.md and execute it. Total runtime, all three together: about fifteen minutes.

09 Demo 1 — Greenfield Bootstrap

Demo one is the greenfield case. The conductor spine is pre-deployed — the agent doesn't scaffold it, it uses it. It reads the active slice, branches, generates eight LookML views from the schema document, writes the model file with eight explores, runs the validator, marks the slice stable, writes a handoff with Exact Next Steps. No scaffolding, no meta-work. First prompt to first commit to first handoff in five minutes.

10 Demo 2 — Pair Programming

Demo two shows the pair programming pattern, which is the core collaboration model. Phase one is fully autonomous — the agent picks up slice-04, generates the new view, adds the ninth explore, validates, hands off. Then phase two: the agent reads the Exact Next Steps from its own handoff, drafts slice-05 as a new spec, commits the draft, and stops. The operator reviews the diff live in the IDE — adjusts scope, sharpens wording — and approves. The agent picks back up and executes.

11 Demo 3 — Automated Maintenance

Demo three simulates a nightly cron job. No human present. The agent wakes up cold, with zero session context, and orients entirely from the conductor spine. Runs the validator. Checks the queue. Reads blockers. Writes a structured status entry — clean or degraded. Does not auto-fix anything. Commits and exits. This is the proof that a well-formed conductor spine is self-describing — any agent, any time, can orient from it alone.

12 Validator

The validator is the governance gate, and it is non-optional. scripts/validate.py is domain-agnostic — it checks spine structure, handoff format, active slice acceptance criteria, CI stub, git branch state. Stdlib only, zero dependencies, CI-safe. Domain-specific checks live as separate extension scripts — validate_lookml.py for this demo. Same pattern works for any domain — copy the extension, replace the checks.

13 Any Agent · Any Project

The scaffold is agent-agnostic. Rename AGENTS.md to match your CLI — that's the only file change needed to switch agents. Same content, just the filename your CLI auto-loads. To adapt for any project: copy the conductor directory, fill in intent.md, edit slice-01, point an agent at the root. The LookML demo is the worked example. The scaffold is the product.

14 Ship It

That's conductor — structured agent workflow for any project. Three demos, three branches, one prompt each: read the DEMO file and execute. The repo is on the screen, MIT licensed, zero dependencies. Clone it, pick a branch, paste the prompt into your agent of choice and watch the loop run. Happy to take questions.

Questions you'll get

→ How is this different from AutoGen / CrewAI / TaskWeaver?

Those are multi-agent orchestration frameworks — they manage agent-to-agent communication at runtime. Conductor is a workflow governance layer — it manages human-to-agent handoffs across sessions. They're not in competition. You could run Conductor to manage the development lifecycle of a system that uses AutoGen internally.

→ Why not just use a well-structured CLAUDE.md?

CLAUDE.md is always-on context — it loads every session whether relevant or not. Conductor is scoped and active: the slice spec tells the agent exactly what this session's job is, the handoff records what actually happened, and the validator enforces it. CLAUDE.md is configuration. Conductor is workflow.

→ Does the agent ever go off-script?

The slice's out-of-scope declarations are instructions, not enforcement. An agent can violate them. That's why the validator and the operator approval step exist — they're the catch. The pattern assumes you review the diff before advancing the queue.

→ What if the handoff is wrong?

The operator reads it before the next session starts. That's the approval step. If the handoff doesn't reflect reality, the operator corrects it before promoting to the next slice. The system doesn't auto-advance — every slice promotion is a human decision.

→ Does this work with local models?

Yes — any agent that can read files and write files. The scaffold has no opinion on what model runs it. Local models tend to be weaker at following multi-step structured instructions, but the format itself is model-agnostic.
